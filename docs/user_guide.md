# NPC Simulator 사용자 안내서

## 1. 이 프로젝트는 무엇인가

`NPC Simulator`는 웹 브라우저에서 실행되는 NPC 상호작용 프로토타입이다.  
사용자는 작은 마을 허브에서 NPC를 선택하고, 말을 걸고, 설득하고, 부탁하고, 정보를 캐낼 수 있다.

이 프로젝트의 핵심은 단순히 NPC가 한 번 답하고 끝나는 챗봇이 아니라는 점이다.

- NPC는 플레이어와의 최근 대화를 기억한다.
- NPC마다 성격, 감정, 현재 목표가 다르다.
- 한 NPC와의 대화 결과가 다른 NPC의 태도나 퀘스트 상태에 영향을 줄 수 있다.
- 왜 그런 답이 나왔는지 감독자 모드에서 내부 판단 근거를 볼 수 있다.

즉, 이 프로젝트는 "말이 통하는 세계 안의 NPC 엔진"을 보여주기 위한 POC다.

## 2. 실행하면 무엇이 보이나

앱을 실행하고 메인 화면에 들어가면 하나의 허브 화면이 보인다.

화면은 크게 세 구역으로 나뉜다.

### 왼쪽: NPC 목록

- 현재 마을 허브에 있는 NPC들이 보인다.
- 각 NPC 카드에는 이름, 역할, 현재 감정, 플레이어와의 관계 요약이 표시된다.
- 클릭하면 중앙 패널의 대상 NPC가 바뀐다.

### 가운데: 선택한 NPC와의 상호작용 영역

- 상단에는 선택한 NPC의 상세 정보가 보인다.
- 현재 성격, 상태, 목표, 관계 수치, 관련 퀘스트가 표시된다.
- 그 아래에는 대화 기록, 자유 입력창, 행동 버튼, 최근 결과 요약이 있다.
- 맨 아래에는 월드 이벤트 로그가 쌓인다.

### 오른쪽: 감독자 모드

- 기본적으로 열려 있다.
- `감독자 닫기` 버튼으로 숨길 수 있다.
- 최근 상호작용에 대해 어떤 기억을 꺼냈는지, 감정이 어땠는지, 무슨 의도를 가졌는지, 어떤 행동 후보를 검토했는지 볼 수 있다.

## 3. 이 앱을 어떻게 사용하나

이 문서는 현재 구현된 프론트엔드 UI 기준으로 설명한다.

### 3-1. 시작하기

프로젝트 루트에서 아래 명령으로 실행한다.

```bash
npm install
npm run dev
```

브라우저에서 로컬 주소를 열면 된다.  
기본 포트는 `3000`이지만 이미 사용 중이면 Next.js가 다른 포트를 자동으로 사용한다.

### 3-2. NPC 선택

1. 왼쪽 `NPC 목록`에서 원하는 NPC를 클릭한다.
2. 중앙 상단 카드가 해당 NPC 정보로 바뀐다.
3. 이 카드에서 현재 감정, 관계 수치, 현재 목표, 관련 퀘스트를 먼저 읽는다.

처음 보는 사람이라면 아래 순서로 접근하는 것이 이해하기 쉽다.

1. `미라(여관 주인)`에게 말을 건다.
2. `도윤(경비병)`으로 바꿔 같은 주제를 물어본다.
3. `엘라(길드 담당자)`에게 가서 의뢰나 신뢰 관련 반응을 확인한다.

### 3-3. 자유 입력으로 대화하기

중앙 하단의 입력창에 자연어로 질문이나 요청을 입력한 뒤 `자유 입력 보내기` 버튼을 누르면 된다.

예시:

- `어제 본 수상한 상인 얘기를 더 해줘`
- `길드 의뢰를 맡으려면 뭘 증명해야 해?`
- `경비 순찰에서 이상한 점이 있었나?`

자유 입력은 가장 자연스럽게 NPC의 기억 반영을 체감하는 방법이다.

### 3-4. 행동 버튼 사용하기

입력창 아래에는 6개의 행동 버튼이 있다.

- `질문`
- `설득`
- `거래`
- `부탁`
- `공감`
- `압박`

이 버튼들은 "같은 주제라도 어떤 태도로 접근하느냐"를 엔진에 전달한다.

예를 들어:

- `질문`은 탐색적 접근이다.
- `설득`은 NPC의 판단을 바꾸려는 시도다.
- `거래`는 정보나 호의를 교환하려는 접근이다.
- `부탁`은 협조 요청에 가깝다.
- `공감`은 긴장을 낮추고 친밀도를 올리는 데 유리하다.
- `압박`은 빠르게 반응을 끌어낼 수 있지만 신뢰를 깎을 수 있다.

현재 UI에서는 행동 버튼을 누를 때 입력창 내용이 함께 있으면 그 문장도 같이 전송된다.  
입력창이 비어 있으면 버튼 자체의 의도만 전송된다.

### 3-5. 결과 읽는 법

상호작용이 끝나면 다음 요소를 같이 보면 된다.

#### 대화 기록

- 플레이어 메시지와 NPC 응답이 차례로 쌓인다.
- 같은 NPC와 다시 대화하면 이전 기록이 문맥으로 반영된다.

#### Last Result

- 방금 대화의 관계 변화량이 보인다.
- `신뢰`, `친밀`, `긴장`이 각각 얼마나 바뀌었는지 표시된다.
- 퀘스트 상태 변화가 있으면 함께 표시된다.

#### 월드 이벤트 로그

- 이번 상호작용이 세계에 어떤 흔적을 남겼는지 요약된다.
- 퀘스트 해금, 태도 변화, 사회적 파장 같은 정보가 쌓인다.

#### 감독자 모드

- NPC가 참고한 기억
- 현재 감정
- 이번 상호작용에서의 의도
- 후보 행동
- 최종 선택 이유

이 5가지를 같이 보면 "왜 저 NPC가 이렇게 말했는지"를 이해할 수 있다.

### 3-6. reset 버튼

상단의 `reset` 버튼을 누르면 현재 상태가 초기 seed 데이터로 되돌아간다.

다음 요소가 초기화된다.

- NPC 상태
- 기억 데이터
- 상호작용 로그
- 퀘스트 상태
- 이벤트 로그
- 감독자 정보

여러 시나리오를 빠르게 다시 테스트할 때 사용하면 된다.

## 4. 처음 사용하는 사람을 위한 추천 체험 순서

이 프로젝트가 무엇을 하는지 가장 빨리 이해하려면 아래 순서를 추천한다.

### 시나리오 A: 기억과 소문 확인

1. `미라`를 선택한다.
2. `어제 본 수상한 상인 얘기를 더 해줘`라고 입력한다.
3. 응답 후 `월드 이벤트 로그`와 `감독자 모드`를 본다.
4. 같은 NPC에게 한 번 더 비슷한 질문을 한다.
5. 이전 대화를 기억한 듯한 반응이 나오는지 확인한다.

### 시나리오 B: NPC 간 파장 확인

1. 위에서 `미라`와 먼저 대화한다.
2. 그 다음 `도윤`으로 바꾼다.
3. `질문` 버튼 또는 `상인`, `창고`, `소문` 관련 입력을 사용한다.
4. 경비병 반응이 달라졌는지, 이벤트 로그나 퀘스트 상태가 이어지는지 확인한다.

### 시나리오 C: 퀘스트 지향 플레이 확인

1. `엘라`를 선택한다.
2. `길드 의뢰를 맡고 싶다`, `내가 뭘 증명해야 하지?` 같은 문장을 입력한다.
3. `부탁`이나 `설득`을 섞어 본다.
4. 퀘스트 카드 상태가 `잠금 -> 가능 -> 진행`으로 바뀌는지 확인한다.

## 5. 내부적으로는 어떻게 동작하나

이 프로젝트는 단순한 프론트엔드 페이지가 아니라, UI와 서버 엔진이 함께 움직이는 구조다.

큰 흐름은 아래와 같다.

1. 화면이 처음 열리면 프론트엔드가 월드 상태를 불러온다.
2. 사용자가 NPC를 선택하고 상호작용을 보낸다.
3. 서버가 해당 NPC의 기억, 관계, 퀘스트 상태를 읽는다.
4. LLM에게 현재 상황을 전달해 NPC 응답과 내부 판단 결과를 받는다.
5. 서버가 관계 변화, 기억 저장, 퀘스트 변화, 이벤트 로그를 갱신한다.
6. 갱신된 월드 상태를 다시 프론트엔드에 반환한다.
7. UI는 새 상태를 바로 다시 렌더링한다.

## 6. 프론트엔드 관점의 동작 원리

현재 프론트엔드는 하나의 허브 화면을 중심으로 동작한다.

### 초기 로드

- 메인 페이지가 열리면 서버에서 현재 월드 스냅샷을 가져온다.
- 이 스냅샷에는 아래 정보가 포함된다.
  - 월드 메타 정보
  - NPC 목록
  - 퀘스트 목록
  - 이벤트 로그
  - NPC별 대화 기록
  - 마지막 감독자 정보
  - 현재 LLM provider 상태

### 상호작용 전송

사용자가 상호작용을 보내면 프론트엔드는 `POST /api/interact`를 호출한다.

전달되는 핵심 값은 아래와 같다.

- 어떤 NPC와 대화하는지
- 자유 입력인지 행동 버튼인지
- 입력한 텍스트
- 어떤 행동 버튼을 눌렀는지
- 플레이어 식별자

응답이 오면 프론트엔드는 아래를 한 번에 갱신한다.

- 대화 기록
- NPC 상태
- 퀘스트 상태
- 이벤트 로그
- 감독자 패널
- 최근 결과 요약

### reset

`reset` 버튼은 `POST /api/reset`을 호출해서 저장된 상태 파일을 seed 상태로 다시 덮어쓴다.

## 7. 서버 엔진 관점의 동작 원리

서버에서 상호작용을 처리할 때는 대략 아래 순서로 진행된다.

### 1. 현재 상태 읽기

서버는 `data/` 폴더의 JSON 파일에서 현재 상태를 읽는다.

- `world-state.json`
- `npc-memory.json`
- `interaction-log.json`

### 2. 입력 정규화

플레이어 입력을 정리한다.

- 자유 입력인지 행동 입력인지 구분
- 행동 버튼 라벨 정리
- LLM에 설명하기 쉬운 문장으로 요약

### 3. 관련 기억 검색

선택한 NPC의 기억 목록에서 현재 입력과 관련성이 높은 기억 몇 개를 뽑는다.

이때 고려하는 것은 대략 아래와 같다.

- 입력 텍스트에 포함된 단어
- 행동 버튼 종류
- 기억 태그
- 기억 중요도
- 최근성
- short / long memory 여부

### 4. LLM 호출

서버는 아래 정보를 묶어서 LLM에 보낸다.

- 플레이어 입력
- NPC의 성격
- 현재 감정
- 현재 관계 상태
- 현재 목표
- 관련 퀘스트
- 최근 이벤트
- 최근 대화 기록
- 방금 검색된 기억

그리고 LLM은 아래 구조의 결과를 돌려준다.

- NPC의 답변 문장
- 현재 감정 요약
- 의도 요약
- 후보 행동 2~3개
- 최종 선택 행동
- 선택 이유

### 5. 비LLM 후처리

LLM 결과를 받은 뒤에는 규칙 기반 로직이 처리한다.

- 관계 수치 변화 계산
- 다른 NPC로의 파장 효과 적용
- 퀘스트 상태 전이
- 새 이벤트 로그 생성
- 새 기억 추가
- 최근 대화 로그 저장

즉, "무슨 말을 할지"는 LLM이 강하게 관여하고, "수치와 상태를 어떻게 바꿀지"는 코드가 관리한다.

### 6. 감독자 정보 생성

같은 상호작용 결과에서 감독자 패널용 데이터도 함께 만든다.

- 어떤 기억을 참고했는지
- 왜 이런 감정이 나왔는지
- 어떤 행동 후보가 있었는지
- 왜 그 행동을 골랐는지

이 정보는 플레이 감각을 해치지 않으면서 내부 엔진을 설명하기 위한 용도다.

## 8. 저장 방식

현재 버전은 DB를 쓰지 않고 파일 기반 저장을 사용한다.

상태는 프로젝트 루트의 `data/` 폴더에 저장된다.

- `world-state.json`: 현재 월드, NPC 상태, 퀘스트, 이벤트, 마지막 감독자 정보
- `npc-memory.json`: NPC별 기억 목록
- `interaction-log.json`: 상호작용 기록

이 방식의 장점은 구조가 단순하고, reset과 디버깅이 쉽다는 점이다.

## 9. LLM provider는 무엇을 의미하나

상단에는 현재 LLM provider 상태가 표시된다.

현재 구현은 두 모드를 지원한다.

### `codex`

- 기본 개발 모드
- `codex CLI` 로그인 상태를 사용한다
- 현재 프로젝트에서 실제 검증이 끝난 경로다

### `openai`

- `OPENAI_API_KEY`가 있어야 한다
- 구조는 구현되어 있지만, 현재 환경에서는 키가 없으면 설정 오류가 나도록 되어 있다

## 10. 이 문서를 읽은 뒤 어디를 보면 좋은가

직접 써보려면 아래 순서가 가장 좋다.

1. 앱을 실행한다.
2. `미라`를 선택한다.
3. 자유 입력으로 소문을 묻는다.
4. `Last Result`와 `월드 이벤트 로그`를 본다.
5. `감독자 모드`를 열어서 내부 판단을 본다.
6. `도윤`, `엘라`로 옮겨 가며 같은 주제가 어떻게 이어지는지 본다.
7. 마지막에 `reset`으로 초기 상태로 되돌린다.

이렇게 해보면 이 프로젝트가 단순한 NPC 대화 UI가 아니라,  
`기억 -> 판단 -> 반응 -> 상태 변화 -> 다음 대화에 반영` 구조를 가진 작은 NPC 시뮬레이션이라는 점을 바로 이해할 수 있다.
