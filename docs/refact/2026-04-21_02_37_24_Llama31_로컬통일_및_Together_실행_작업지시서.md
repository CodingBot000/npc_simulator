# Llama 3.1 로컬 통일 및 Together Serverless LoRA 실행 작업지시서

작성 시각: 2026-04-21 02:37:24 KST  
대상 브랜치: `refact/peft_canonical_mlx_derivatives`

## 문서 목적

이 문서는 새 Codex 창에서 그대로 이어받아 개발할 수 있도록, 현재 저장소 상태와 다음 작업 순서를 누락 없이 정리한 작업지시서다.

이번 작업의 최종 목표는 아래 두 가지를 모두 만족하는 것이다.

1. 로컬 학습 및 로컬 MLX 런타임 경로를 `Qwen` 기준에서 `Llama 3.1` 기준으로 통일한다.
2. 같은 데이터셋을 재사용하여 `Together serverless LoRA` 경로로 실제 remote fine-tuning 1회를 성공시킨다.

중요:

- 이번 단계의 대상은 `SFT`다.
- `Together + DPO`는 이번 단계에서 지원하지 않는다.
- `Qwen adapter`를 `Llama`에 재활용하는 작업은 하지 않는다.
- `dedicated endpoint`는 만들지 않는다.
- 목표는 `serverless LoRA`다.

---

## 현재 상태 요약

현재 저장소에는 이미 아래 큰 흐름이 구현되어 있다.

- `PEFT canonical + MLX runtime derivative` 구조가 들어가 있다.
- `Together serverless LoRA`용 remote training backend가 코드상 추가되어 있다.
- `trainingBackend`, `remoteJobId`, `remoteModelName` 등 remote training metadata 저장 구조가 들어가 있다.
- `Golden-set Evaluation`도 local artifact뿐 아니라 remote model 후보를 평가할 수 있게 일부 확장되어 있다.

즉, 완전 처음부터 시작하는 작업이 아니다.  
이번 작업은 `Qwen 중심 기본값과 경로를 Llama 3.1 기준으로 재정렬`하고, 그 상태로 `실제 Together remote fine-tune`까지 검증하는 마무리 단계에 가깝다.

## 이번 작업의 최종 완료 정의

이번 작업은 아래가 모두 만족되면 완료로 본다.

1. 로컬 SFT 기본 base가 더 이상 `Qwen`이 아니라 `Llama 3.1` 계열이다.
2. 로컬 MLX 런타임 기본 base도 `Llama 3.1` 계열이다.
3. 로컬 real SFT 1회를 돌려 `adapter_config.json + adapter_model.safetensors` 생성이 확인된다.
4. 생성된 canonical adapter에서 MLX runtime 파생본을 만들고 로컬 추론까지 성공한다.
5. `Together serverless LoRA` 경로에서 같은 finalize 데이터셋으로 remote fine-tune job 생성이 성공한다.
6. 완료된 run에 `remoteJobId`, `remoteTrainingFileId`, `remoteValidationFileId`, `remoteModelName`가 저장된다.
7. `remoteModelName`으로 Together inference 1회가 성공한다.
8. 결과와 트러블슈팅을 `docs/progress.md`에 이어서 기록한다.

## 이번 작업에서 사용할 모델 분리 원칙

이 작업에서 가장 중요한 것은 `base model`을 하나로 뭉뚱그리지 않고 역할별로 분리하는 것이다.

반드시 아래 3가지를 구분해서 다뤄라.

1. 로컬 canonical PEFT training base
2. 로컬 MLX runtime base
3. Together remote training base

이 세 값은 논리적으로 같은 계열이어야 하지만, 실제 모델 ID 문자열은 다를 수 있다.

예상 방향:

- 로컬 canonical PEFT training base: `Meta-Llama-3.1-8B-Instruct` 계열
- 로컬 MLX runtime base: `mlx-community`의 Llama 3.1 8B Instruct 4bit 계열
- Together remote training base: `meta-llama/Meta-Llama-3.1-8B-Instruct-Reference`

주의:

- 위 문자열은 시작점일 뿐이다.
- 새 창에서 실제로 사용할 HF/MLX repo id는 접근 가능성, gating, tokenizer 호환성을 보고 확정해야 한다.
- 특히 `Llama` 계열은 HF access grant 또는 라이선스 승인 여부가 변수다.

## 1단계. base model 설정 구조를 역할별로 분리

목표:

- 지금 코드에 남아 있는 `Qwen 기본값`을 없애고, local/remote 목적별 base model 설정을 분리한다.

왜 필요한가:

- 현재는 `CANONICAL_TRAINING_BASE_MODEL` 하나로 local/remote 문맥이 섞일 가능성이 있다.
- 이번 작업은 `로컬도 Llama`, `Together도 Llama`로 맞추되 모델 ID가 완전히 같을 필요는 없다.

수정 대상 후보:

- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
- `backend/scripts/review/training.ts`
- `backend/scripts/server/config.ts`
- 로컬 reply adapter 관련 기본값 사용처 전부
- 문서에 박혀 있는 `Qwen/Qwen2.5-7B-Instruct`

해야 할 일:

1. 아래 환경변수 구조로 정리하는 방향을 우선 검토한다.
   - `LOCAL_CANONICAL_TRAINING_BASE_MODEL`
   - `LOCAL_REPLY_MLX_MODEL`
   - `REMOTE_TRAINING_BASE_MODEL`
2. 기본값은 Llama 3.1 기준으로 맞춘다.
3. Together mode에서는 `REMOTE_TRAINING_BASE_MODEL`을 사용하게 한다.
4. Local PEFT mode에서는 `LOCAL_CANONICAL_TRAINING_BASE_MODEL`을 사용하게 한다.
5. 로컬 reply/runtime 경로는 `LOCAL_REPLY_MLX_MODEL`을 사용하게 한다.

주의:

- `ReviewService`는 Java env를 읽고
- `training.ts`/`together-client.ts`는 Node env 또는 `.env.local`을 읽는다.
- 두 런타임이 env를 다르게 읽는 점을 문서와 코드에 분명히 반영해야 한다.

완료 기준:

- 코드상 기본 base가 더 이상 Qwen이 아님
- local canonical / local runtime / remote training base를 분리 설명할 수 있음

## 2단계. 로컬 SFT 경로를 Llama 기준으로 전환

목표:

- finalize 데이터셋으로 Llama base 기반 canonical PEFT adapter를 실제 생성한다.

수정 및 확인 대상:

- `backend/scripts/train-peft-sft.py`
- `backend/scripts/review/training.ts`
- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
- dataset exporter 관련 스크립트
- training manifest 작성 로직

해야 할 일:

1. local SFT run의 base model이 Llama로 들어가도록 확인한다.
2. 기존 finalize dataset 재사용 경로는 유지한다.
3. 필요하면 local용 테스트 run id를 수동으로 하나 만든다.
4. real SFT 1회를 실행한다.

실행 기준:

- 입력은 이미 finalize된 SFT 데이터셋을 사용
- 목적은 품질학습이 아니라 파이프라인 검증

기대 산출물:

- `canonical/adapter_config.json`
- `canonical/adapter_model.safetensors`
- `training-result.json`

중단 조건:

- HF access 부족으로 base model 다운로드 실패
- tokenizer/chat template 불일치로 학습 시작 자체가 막힘
- MPS/메모리 이슈로 시작 불가

중단 시 해야 할 일:

- 에러 메시지를 문서화
- 필요한 모델 ID 수정 또는 quantization 전략 조정
- 무작정 범위를 넓히지 말고 local SFT 1회 성공을 먼저 만든다

완료 기준:

- Llama base로 canonical PEFT adapter가 실제 생성됨

## 4단계. Together remote SFT 경로 마무리 점검

목표:

- dataset export -> file upload -> remote fine-tune job -> polling -> output_name 저장 경로를 실제 실행 가능한 상태로 마무리한다.

이미 구현된 것으로 가정하는 파일:

- `backend/scripts/server/together-client.ts`
- `backend/scripts/export-together-sft-dataset.mjs`
- `backend/scripts/review/training.ts`
- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
- `backend/src/main/resources/db/migration/V6__add_training_remote_backend_columns.sql`

새 창 Codex가 다시 확인해야 할 핵심:

1. `runTraining` 호출 시 `kind=sft`만 보내면 Together mode에서 remote path로 가는지
2. `ReviewService`의 `TRAINING_EXECUTION_MODE=together_serverless_lora`가 실제로 적용되는지
3. training worker가 build 후 local train으로 빠지지 않고 `runTogetherTrainingWorker(...)`로 들어가는지
4. Together file upload 응답 파싱이 현재 API 응답과 맞는지
5. fine-tune status polling과 event 저장이 정상인지
6. 성공 시 `remoteModelName`이 DB와 view에 반영되는지

완료 기준:

- 코드 경로상 Together mode가 실제 remote job 생성으로 이어질 준비가 끝남

## 5단계. Together 실제 fine-tune 1회 실행

목표:

- 실제 Together API를 사용해 SFT remote fine-tuning job 1회를 성공시킨다.

실행 전 조건:

1. 루트 `.env.local`에 `TOGETHER_API_KEY`가 존재해야 함
2. 필요하면 `HF_TOKEN`도 존재해야 함
3. backend를 Together mode로 실행해야 함

중요:

- Node helper는 `.env.local`을 읽을 수 있어도
- Java `ReviewService`는 쉘 env 또는 실행 설정값이 필요할 수 있다

권장 실행 예시:

```bash
cd /Users/switch/Development/Web/npc_simulator/backend

export TRAINING_EXECUTION_MODE=together_serverless_lora
export REMOTE_TRAINING_BASE_MODEL=meta-llama/Meta-Llama-3.1-8B-Instruct-Reference

./gradlew bootRun
```

실행 방법:

1. finalize가 완료된 상태에서 UI 또는 API로 `POST /api/review/training` 호출
2. body는 `{"kind":"sft"}` 사용
3. active run 상태를 polling
4. step이 아래 순서로 가는지 확인
   - `build_dataset`
   - `upload_remote_files`
   - `wait_remote_training`

기대 결과:

- `remoteTrainingFileId`
- `remoteValidationFileId`
- `remoteJobId`
- `remoteModelName`

실패 시 점검 순서:

1. Together credits 부족
2. base model 미지원
3. dataset 형식 불일치
4. API schema mismatch
5. env 주입 실패로 Together mode가 실제 적용되지 않음

완료 기준:

- Together remote fine-tune job이 성공 상태가 됨
- run 상세에 `remoteModelName`까지 저장됨

## 6단계. Together remote inference 실제 확인

목표:

- fine-tune 완료 model의 `output_name`으로 실제 chat completion 1회를 성공시킨다.

해야 할 일:

1. `remoteModelName`을 확인한다.
2. `backend/scripts/server/together-client.ts` 경로 또는 직접 API 호출로 inference를 1회 한다.
3. 성공 응답 텍스트를 캡처한다.

가능한 확인 방식:

```bash
curl https://api.together.xyz/v1/chat/completions \
  -H "Authorization: Bearer $TOGETHER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "여기에 remoteModelName",
    "messages": [
      {"role": "user", "content": "짧게 자기소개해봐"}
    ],
    "max_tokens": 128
  }'
```

완료 기준:

- remoteModelName으로 실제 응답이 돌아옴

## 새 창 Codex가 반드시 기억할 운영상 사실

1. Together에서 지금 원하는 건 `full model upload`가 아니다.
2. 원하는 것은 `Llama base + LoRA fine-tuning + serverless inference`다.
3. 기존 `Qwen adapter`는 재사용 대상이 아니다.
4. `로컬도 Llama`, `Together도 Llama`로 맞추는 것이 이번 작업 핵심이다.
5. `serverless`가 목표이므로 `dedicated endpoint`는 만들지 않는다.
6. `DPO`는 이번 단계에서 건드리지 않는 것이 맞다.

## 최종 인계 문구

새 창 Codex는 이 문서를 기준으로 아래 순서만 따르면 된다.

1. 현재 Together 연동 변경분을 보존하고 이해한다.
2. base model 구조를 local canonical / local runtime / remote training으로 분리한다.
3. 로컬 기본값을 Qwen에서 Llama로 바꾼다.
4. local real SFT를 성공시킨다.
5. local MLX runtime derive와 inference를 성공시킨다.
6. Together remote SFT fine-tune을 실제로 1회 실행한다.
7. remoteModelName으로 inference를 확인한다.
8. 결과를 `docs/progress.md`에 기록한다.

이번 단계의 성공 기준은 `Llama 기준으로 local + Together가 모두 한 번씩 실제로 돈 상태`다.
