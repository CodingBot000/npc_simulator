# PEFT 정본 + MLX 파생본 전환 작업지시서

작성 시각: 2026-04-20 03:05:37

## 문서 목적

이 문서는 현재 MLX 중심 학습/서빙 구조를 `PEFT 정본 + MLX 파생본` 구조로 전환하기 위한 실행 문서다.

이 문서는 두 용도로 쓴다.

1. Codex가 실제 구현 순서와 완료 조건을 놓치지 않고 개발하기 위한 작업지시서
2. 사람이 현재 구조, 바꿀 구조, 리스크, 검증 기준을 한 번에 이해하기 위한 설명 문서

이 문서는 아이디어 메모가 아니라 **실행 순서가 고정된 개발 문서**다.  
가능한 한 “어디를 왜 바꾸는지”와 “어느 단계에서 무엇을 검증해야 하는지”를 명시한다.

---

## 이번 작업의 최종 목표

이번 전환 작업의 최종 목표는 아래 다섯 가지다.

1. 학습의 정본 산출물을 `HF PEFT LoRA adapter`로 바꾼다.
2. 로컬 Mac 런타임용 산출물은 정본에서 파생한 `MLX runtime artifact`로 관리한다.
3. `Together / vLLM / TGI` 같은 외부 스택에는 정본 adapter를 재사용할 수 있게 만든다.
4. 로컬 런타임과 Golden-set Evaluation은 MLX 파생본만 참조하게 만든다.
5. training metadata, promotion, UI, smoke test가 모두 이 이중 산출물 구조를 이해하게 만든다.

한 줄로 요약하면 다음과 같다.

- 학습/배포의 기준점은 `PEFT adapter`
- 로컬 Mac 실행의 기준점은 `MLX 파생본`

---

## 이번 작업에서 명시적으로 달성해야 하는 산출물

이 작업이 끝나면 최소한 아래 상태가 되어야 한다.

1. 성공한 SFT run은 `canonical PEFT adapter`와 `runtime MLX artifact`를 둘 다 남긴다.
2. 성공한 DPO run도 같은 구조를 따른다.
3. DPO는 직전 SFT run의 `canonical adapter`를 parent로 사용한다.
4. runtime provider는 더 이상 `output_adapter_path` 하나만 믿지 않고 `runtime artifact`를 사용한다.
5. eval worker도 후보/베이스라인 비교 시 `runtime artifact`를 사용한다.
6. review dashboard는 run 상세에서 `canonical path`와 `runtime path`를 구분해 보여준다.
7. 기존 legacy MLX adapter run도 즉시 깨지지 않도록 fallback이 있다.

---

## 비목표

이번 작업에서 하지 않을 일도 명확히 적는다.

1. Together 업로드 자동화까지 한 번에 끝내지 않는다.
2. 모든 historical run 메타데이터를 강제로 재마이그레이션하지 않는다.
3. MLX adapter 직접 변환 경로가 불안정하면 거기에 집착하지 않는다.
4. 로컬 Mac에서 실학습까지 반드시 계속 가능하게 만들겠다는 목표를 두지 않는다.
5. runtime 전체를 cloud serving으로 바로 교체하지 않는다.

이번 작업의 핵심은 “범용 정본 확보 + 로컬 파생본 분리”다.  
배포 자동화는 그 다음 단계다.

---

## 현재 상태 요약

현재 코드는 `학습 결과 = MLX adapter directory 하나`라는 전제를 깔고 있다.

대표 근거는 다음과 같다.

1. `backend/scripts/review/training.ts`
   - SFT는 `.venv/bin/mlx_lm.lora`를 직접 실행한다.
   - DPO는 `backend/scripts/train-mlx-dpo.py`를 직접 실행한다.
   - run 완료 시 `adapter_output` 하나만 등록한다.
2. `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
   - preflight, training run spec, promotion, eval 기준이 모두 `output_adapter_path` 중심이다.
3. `backend/scripts/server/providers/mlx-reply-adapter.ts`
   - `adapters.safetensors`가 있는 디렉터리를 runtime adapter로 간주한다.
4. `backend/scripts/review/eval.ts`
   - 후보/베이스라인 모두 `output_adapter_path`를 바로 MLX generation에 넣는다.
5. `frontend/src/lib/review-types.ts`
   - training run view에 `adapterPath` 단일 필드만 있다.
6. `frontend/src/components/review/review-dashboard.tsx`
   - run 상세에 `adapter` 하나만 표시한다.
7. `backend/src/test/java/com/npcsimulator/api/controller/ReviewTrainingSmokeIntegrationTests.java`
   - `output_adapter_path` 디렉터리 하나와 그 내부 파일만 검증한다.

즉 지금 구조는 다음 가정을 암묵적으로 갖고 있다.

- 정본 산출물과 로컬 실행 산출물이 같다.
- DPO parent와 runtime artifact가 같다.
- promotion 대상과 deployment 대상이 같다.

이번 작업은 이 전제를 깨고, 산출물을 둘로 분리하는 작업이다.

## 핵심 설계 결정

### 1. 정본 base model은 HF ID를 사용한다

정본 학습의 기준 base model은 `Qwen/Qwen2.5-7B-Instruct`로 둔다.

중요:

- 현재 로컬 MLX 기본값은 `mlx-community/Qwen2.5-7B-Instruct-4bit`다.
- 이것은 로컬 실행용 모델 참조값이지, canonical training base로 쓰기엔 맞지 않는다.
- 따라서 기존 `LOCAL_TRAINING_MODEL`과 `LOCAL_REPLY_MLX_MODEL`을 분리해야 한다.

필수 신규 개념:

- `CANONICAL_TRAINING_BASE_MODEL=Qwen/Qwen2.5-7B-Instruct`
- `LOCAL_REPLY_MLX_MODEL=mlx-community/Qwen2.5-7B-Instruct-4bit`

### 2. 정본 산출물은 PEFT adapter다

정본 산출물은 아래 형식을 기준으로 한다.

- `adapter_config.json`
- `adapter_model.safetensors`

여기서 말하는 정본은 `Together에 base model 위에 붙일 수 있는 adapter`를 뜻한다.

### 3. 로컬 런타임 산출물은 MLX 파생본이다

로컬 런타임 산출물은 `canonical PEFT adapter`에서 파생한 MLX용 artifact다.

여기서 가장 중요한 점:

- MLX 파생본이 반드시 `MLX adapter`일 필요는 없다.
- 우선순위는 `안정적으로 로컬에서 실행되는 것`이다.

이번 작업의 기본값은 아래처럼 잡는다.

1. 1순위: `MLX fused model directory`
2. 예외적 대안: deterministic 변환이 확인될 경우 `MLX adapter directory`

즉 이번 작업 문맥에서 `MLX 파생본`은 개념적으로는 넓고, 구현 기본값은 `fused model`이다.

### 4. training run은 canonical과 runtime을 둘 다 기록한다

DB와 read model은 아래를 구분해야 한다.

1. canonical adapter path
2. runtime artifact path
3. runtime artifact kind
4. base model id

이번 전환에서 `output_adapter_path`는 **canonical adapter path** 의미로 정리한다.

이유:

- DPO parent로 재사용해야 하는 쪽이 canonical이다.
- Together/vLLM/TGI로 넘길 수 있는 쪽도 canonical이다.
- 이름상 `adapter path`는 runtime fused model보다 canonical 쪽에 더 잘 맞는다.

대신 runtime용 경로는 별도 필드로 뺀다.

권장 신규 필드:

- `runtime_artifact_path`
- `runtime_artifact_kind`
- `base_model_id`

### 5. runtime / eval / promotion은 runtime artifact를 사용한다

이후 로컬 런타임과 Golden-set Evaluation은 canonical adapter가 아니라 runtime artifact를 읽어야 한다.

즉 책임 분리는 다음과 같다.

- 학습/재사용/배포 기준: canonical adapter
- 로컬 Mac inference 기준: runtime artifact

### 6. legacy run fallback을 반드시 둔다

historical run에는 새 필드가 없을 것이다.  
따라서 아래 fallback을 반드시 유지한다.

1. `runtime_artifact_path`가 있으면 그걸 우선 사용
2. 없으면 `output_adapter_path`가 MLX legacy adapter인지 확인
3. legacy adapter면 기존 방식으로 실행
4. 둘 다 없으면 runtime/eval에서 명시적 오류

이 fallback이 없으면 이미 promotion 된 과거 run이 깨질 수 있다.

---

## 목표 아키텍처

전환 후 개념 구조는 아래와 같다.

1. review finalize가 dataset snapshot을 만든다.
2. training orchestration이 canonical training을 실행한다.
3. canonical training은 `PEFT adapter`를 만든다.
4. derivation step이 canonical adapter에서 `MLX runtime artifact`를 만든다.
5. run metadata는 둘 다 저장한다.
6. DPO는 canonical adapter를 parent로 이어받는다.
7. promotion은 run 자체를 promotion하지만, runtime 쪽은 `runtime artifact`를 사용한다.
8. 향후 Together 업로드 시에는 canonical adapter만 사용한다.

---

## 구현 원칙

1. 실학습 전환보다 먼저 데이터 계약을 고정한다.
2. runtime을 먼저 깨뜨리지 않는다.
3. DPO parent 경로가 canonical로 바뀌지 않으면 작업을 완료로 보지 않는다.
4. MLX 파생본 형식이 불명확하면 `fused model`을 기본값으로 채택한다.
5. smoke mode와 legacy run은 마지막까지 유지한다.
6. UI 문구 정리는 구조 안정화 다음이 아니라, 같은 작업 안에서 같이 정리한다.

## 단계별 실행 계획

## 0단계. 선행 스파이크와 형식 검증

목적:

- PEFT adapter에서 로컬 runtime artifact를 어떤 방식으로 만들지 고정한다.

필수 작업:

1. 작은 샘플 adapter 기준으로 `PEFT adapter -> merged HF model -> MLX artifact` 경로를 검증한다.
2. 파생본 형식을 `mlx_fused_model` 또는 `mlx_adapter` 중 하나로 고정한다.
3. 실패 시 fallback 기준을 문서화한다.

권장 구현:

- 새 스크립트 추가
  - `backend/scripts/train-peft-sft.py`
  - `backend/scripts/train-peft-dpo.py`
  - `backend/scripts/derive-mlx-runtime-from-peft.py`

0단계 완료 조건:

1. 샘플 canonical adapter를 입력으로 받아 runtime artifact를 만드는 최소 스크립트가 존재한다.
2. 생성된 runtime artifact를 로컬에서 한 번 호출해 응답을 받았다.
3. runtime artifact kind를 `mlx_fused_model` 또는 `mlx_adapter` 중 하나로 확정했다.

중요:

이 단계가 끝나기 전까지는 control plane 대규모 수정에 들어가지 않는다.

## 2단계. SFT real training을 PEFT canonical training으로 교체

목적:

- SFT 정본 생성 경로를 MLX에서 PEFT로 바꾼다.

필수 작업:

1. 현재 `.venv/bin/mlx_lm.lora` 호출 제거
2. canonical training base를 `Qwen/Qwen2.5-7B-Instruct`로 사용
3. 산출물을 `outputs/training/<run>/canonical/`에 저장
4. `training-result.json`에 canonical/runtime metadata를 기록

권장 디렉터리 구조:

- `outputs/training/<run>/canonical/`
- `outputs/training/<run>/runtime/`
- `outputs/training/<run>/training-result.json`

필수 수정 파일:

- `backend/scripts/review/training.ts`
- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
- 신규 `backend/scripts/train-peft-sft.py`

필수 env 분리:

- `CANONICAL_TRAINING_BASE_MODEL`
- `LOCAL_REPLY_MLX_MODEL`

주의:

- 기존 `TRAINING_BASE_MODEL = LOCAL_TRAINING_MODEL || appConfig.localReply.mlxModel` 구조는 제거해야 한다.
- canonical training이 MLX repo id를 읽지 않도록 해야 한다.

2단계 완료 조건:

1. 성공한 SFT run에서 canonical adapter 파일이 생성된다.
2. run metadata의 `output_adapter_path`는 canonical path를 가리킨다.
3. `base_model_id`가 저장된다.

---

## 3단계. DPO real training을 PEFT canonical training 기준으로 교체

목적:

- DPO parent가 runtime artifact가 아니라 canonical SFT adapter를 참조하도록 만든다.

필수 작업:

1. latest successful SFT run 조회 시 canonical path를 parent로 사용
2. DPO trainer를 PEFT/TRL 기반으로 교체
3. DPO 성공 후에도 canonical + runtime artifact를 둘 다 생성

필수 수정 파일:

- `backend/scripts/review/training.ts`
- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`
- 신규 `backend/scripts/train-peft-dpo.py`

중요:

현재 preflight는 latest successful SFT의 `output_adapter_path` 하나만 보여준다.  
이 단계에서 이 의미를 canonical path 기준으로 정리해야 한다.

3단계 완료 조건:

1. DPO가 canonical SFT adapter를 parent로 사용한다.
2. DPO run 결과도 canonical/runtime dual artifact를 남긴다.
3. DPO preflight와 status 응답이 새 의미와 맞는다.

---

## 4단계. MLX runtime artifact derivation을 training flow 안에 편입

목적:

- 학습 성공 직후 로컬 실행용 artifact를 자동으로 만든다.

필수 작업:

1. canonical training 성공 뒤 derivation step 실행
2. runtime artifact path/kind 저장
3. training-result manifest 기록
4. 실패 시 canonical 성공 / runtime derivation 실패를 구분해 표시

권장 `training-result.json` 예시:

```json
{
  "runId": "2026-04-20T03-05-37-000000Z_sft",
  "baseModelId": "Qwen/Qwen2.5-7B-Instruct",
  "canonicalArtifact": {
    "kind": "peft_adapter",
    "path": "/abs/outputs/training/<run>/canonical"
  },
  "runtimeArtifact": {
    "kind": "mlx_fused_model",
    "path": "/abs/outputs/training/<run>/runtime"
  },
  "derivation": {
    "status": "succeeded"
  }
}
```

필수 작업 세부:

1. derivation 단계는 train step과 분리된 로그 구간을 남긴다.
2. runtime artifact가 실패하면 run 전체를 실패로 처리할지, degraded success로 둘지 결정해야 한다.
3. 이번 작업의 기본 정책은 `runtime derivation 실패 = run failed`로 둔다.

이유:

- 현재 로컬 eval/promotion/runtime은 runtime artifact가 없으면 바로 못 쓴다.
- control plane에서는 “학습은 끝났는데 쓸 수 없다”보다 “실패”가 더 명확하다.

4단계 완료 조건:

1. 성공 run마다 runtime artifact가 자동 생성된다.
2. artifact table에 canonical/runtime/manifest가 모두 등록된다.
3. `training-result.json` 하나만 봐도 run 산출물이 이해된다.

---

## 5단계. runtime provider / eval worker / promotion read path cutover

목적:

- 로컬 실행 계층이 canonical path를 직접 읽지 않도록 만든다.

필수 작업:

1. runtime provider가 `runtime_artifact_path`와 `runtime_artifact_kind`를 우선 사용
2. eval worker가 candidate/baseline을 runtime artifact 기준으로 비교
3. promotion 이후 runtime selection도 runtime artifact 기준으로 동작

필수 수정 파일:

- `backend/scripts/server/providers/mlx-reply-adapter.ts`
- `backend/scripts/review/eval.ts`
- `backend/src/main/java/com/npcsimulator/review/ReviewService.java`

세부 지시:

1. `mlx-reply-adapter.ts`
   - `adapters.safetensors` 존재 가정 제거
   - artifact kind가 `mlx_adapter`면 `--adapter-path`
   - artifact kind가 `mlx_fused_model`이면 `--model <runtime path>` 사용
   - legacy fallback 유지
2. `review/eval.ts`
   - baseline/candidate resolve 시 runtime artifact 우선
   - generation helper가 adapter/fused model 둘 다 처리
3. Java service
   - promotion status 응답에 canonical/runtime 둘 다 노출

5단계 완료 조건:

1. promoted 최신 run이 새 runtime artifact로 실제 응답을 만든다.
2. Golden-set Evaluation이 새 runtime artifact로 정상 동작한다.
3. legacy MLX adapter promoted run도 fallback으로 여전히 동작한다.

---

## 6단계. review dashboard / API 응답 / operator visibility 정리

목적:

- 운영자가 canonical/runtime dual artifact 구조를 UI에서 볼 수 있게 만든다.

필수 작업:

1. run 상세에 canonical path, runtime path, runtime kind 표시
2. preflight에 DPO parent canonical path를 구분해서 표기
3. 에러 메시지에서 “adapter 없음”과 “runtime artifact 없음”을 구분

필수 수정 파일:

- `frontend/src/lib/review-types.ts`
- `frontend/src/components/review/review-dashboard.tsx`

UI에서 반드시 보여야 하는 값:

1. base model id
2. canonical artifact path
3. runtime artifact path
4. runtime artifact kind
5. derivation status

6단계 완료 조건:

2. current action run / latest run / historical run 모두 새 필드를 보여준다.

## 완료 판정 기준

이번 작업은 아래 조건을 모두 만족해야 완료다.

1. 새 SFT run이 canonical PEFT adapter를 남긴다.
2. 새 SFT run이 runtime MLX artifact도 남긴다.
3. 새 DPO run이 canonical SFT adapter를 parent로 사용한다.
4. promoted run이 runtime artifact로 실제 로컬 응답을 생성한다.
5. dashboard에서 canonical/runtime dual artifact가 보인다.
6. smoke/integration test가 새 구조 기준으로 통과한다.
7. legacy promoted run 최소 1개는 fallback으로 계속 동작한다.

이 중 하나라도 빠지면 이 작업은 끝난 것이 아니다.

## 리스크와 대응

### 리스크 1. PEFT -> MLX adapter 직접 변환이 불안정

대응:

- 기본 경로를 `mlx_fused_model`로 둔다.
- adapter hot-swap 로컬 유연성보다 실행 안정성을 우선한다.

### 리스크 2. runtime/eval이 새 artifact를 못 읽어 historical run이 깨짐

대응:

- legacy fallback을 마지막까지 유지한다.
- historical promoted run으로 수동 검증한다.

### 리스크 3. DPO parent가 실수로 runtime artifact를 가리킴

대응:

- DPO preflight와 run spec에서 parent 필드를 explicit canonical path로 분리한다.
- 관련 테스트를 추가한다.

### 리스크 4. smoke mode가 구조 변경을 반영 못 해 테스트만 통과하고 실환경이 깨짐

대응:

- smoke 산출물도 canonical/runtime dual artifact를 흉내 내게 만든다.
- smoke test assertion을 경로 수가 아니라 artifact 의미 기준으로 바꾼다.

## 최종 한 줄 결론

이번 작업의 본질은 `MLX 하나로 모든 걸 해결`하는 것이 아니라,  
`PEFT adapter를 범용 정본으로 고정하고, 로컬 Mac 실행만 MLX 파생본으로 분리`하는 것이다.

즉 앞으로의 기준은 다음과 같다.

- 재사용/배포 기준: `PEFT canonical adapter`
- 로컬 실행 기준: `MLX derived runtime artifact`
