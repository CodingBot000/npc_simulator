# 자동 NPC 상호작용 V1 작업지시서

## 문서 목적

이 문서는 펠라지아-9 시나리오에 `플레이어 턴 이후 NPC들끼리도 상황에 따라 판세를 흔드는 자동 상호작용 phase`를 추가하기 위한 구현 지시서다.

이 문서는 곧바로 개발에 들어갈 수 있는 수준의 요구사항, 역할 분리, 데이터 구조, 랜덤 정책, 테스트 기준을 포함한다.
향후 문서화와 운영 가이드를 고려해 용어와 구조를 안정적으로 유지하는 것을 전제로 한다.

## 핵심 결정

1. 이 기능은 `LLM 없이` 구현한다.
2. 이 기능은 `규칙 + 가중 랜덤` 기반으로 구현한다.
3. 같은 플레이 입력과 같은 액션 순서를 반복해도 `새 에피소드에서는 결과가 같아지지 않아야 한다`.
4. 다만 디버그와 테스트를 위해 `고정 시드 모드`는 별도로 지원할 수 있다. 기본 게임 플레이에서는 사용하지 않는다.
5. 구현은 기존 플레이어 턴 로직에 억지로 끼워 넣지 말고, `NPC autonomy 전용 모듈`로 분리한다.

## 문제 정의

현재 턴 구조는 사실상 다음과 같다.

1. 플레이어가 NPC 한 명에게 말하거나 행동한다.
2. 그 NPC의 반응과 압력 변화가 계산된다.
3. 라운드가 진행되고 종료 판정이 계산된다.

이 구조의 한계는 다음과 같다.

- 방 안의 다른 NPC들이 너무 정적이다.
- 플레이어가 누르지 않으면 판세가 멈춰 있는 것처럼 보인다.
- 동일한 패턴의 입력이 반복될 때 체감 결과가 단조롭다.
- “각자 살아남으려고 계산 중인 밀실”이라는 테마에 비해 방 자체의 자율성이 약하다.

## 목표

### 제품 목표

- 플레이어 턴이 끝날 때 방 안의 다른 NPC들이 소규모 후속 반응을 하도록 만든다.
- 플레이어가 하지 않은 말이라도, 이미 형성된 압력과 라운드 이벤트 때문에 `판세가 스스로 조금씩 흔들리는 느낌`을 만든다.
- 매번 결과가 조금씩 달라져서 플레이 감각이 더 살아나게 한다.

### 구현 목표

- 플레이어 턴과 NPC 자동 반응 턴을 코드상 명확히 분리한다.
- 랜덤이 들어가더라도 디버깅과 회귀 테스트가 가능하도록 RNG를 모듈화한다.
- V1은 작고 설명 가능한 상태 변화만 만든다.

## 비목표

아래 항목은 V1 범위에서 제외한다.

- NPC 자동 상호작용에 LLM 호출 사용
- NPC 자동 상호작용에 실제 대사 생성
- NPC 간 기억 시스템을 플레이어 메모리 수준으로 확장
- NPC 자동 반응이 플레이어 신뢰/호감/긴장을 직접 바꾸는 것
- 여러 NPC가 길게 연쇄 대화를 이어가는 시스템
- 프론트에서 별도 타임라인 리플레이 UI 구축

## 용어 정의

문서화 일관성을 위해 아래 용어를 고정한다.

- `Autonomy Phase`
  플레이어 턴 직후 자동으로 실행되는 NPC 후속 반응 단계
- `Autonomy Step`
  Autonomy Phase 안에서 발생한 단일 NPC 반응 1회
- `Autonomy Move`
  NPC가 취한 자동 반응 유형
- `Live Seed`
  실제 플레이에서 에피소드마다 새로 생성되는 랜덤 시드
- `Debug Seed`
  테스트/검증용 고정 시드

## 사용자 체감 요구사항

1. 같은 플레이 시퀀스를 새로 시작해서 반복해도 결과가 늘 같아 보이면 안 된다.
2. 그러나 결과가 완전히 무작위처럼 튀면 안 된다.
3. 플레이어가 만든 큰 방향은 유지하되, 세부 판세만 조금씩 달라져야 한다.
4. 자동 반응은 플레이어 행동보다 `작은 규모`로 움직여야 한다.
5. 이벤트 로그를 보면 “왜 흔들렸는지”를 짐작할 수 있어야 한다.

## 기능 요구사항

### FR-1. 턴 후 자동 반응 실행

- 플레이어 턴이 끝난 뒤 `Autonomy Phase`를 실행한다.
- 기본적으로 한 턴에 `1~2개`의 Autonomy Step만 발생하게 한다.
- 기본값은 1개, 특정 조건에서 2개까지 허용한다.

### FR-2. 자동 반응은 NPC끼리의 판세에만 개입

- V1에서 자동 반응은 `NPC 대 NPC` 구도만 흔든다.
- 플레이어 관련 직접 수치인 `playerTrust`, `playerAffinity`, `playerTension`은 V1에서 건드리지 않는다.
- 대신 아래 상태는 바꿀 수 있다.
  - `judgements`
  - `npc.relationship.npcOpinions`
  - `npc.emotion`
  - 필요하면 `npc.statusLine`은 템플릿 수준에서만 경미하게 반영 가능

### FR-3. 자동 반응은 설명 가능해야 함

- 모든 Autonomy Step은 템플릿 기반 이벤트 로그 한 줄로 남긴다.
- 예시
  - `박도현이 다시 마야 로웰 쪽 책임선을 밀었다.`
  - `한유리가 서진호 쪽으로 몰리던 시선을 잠시 주저하게 만들었다.`

### FR-4. 결과는 새 에피소드마다 달라져야 함

- 동일한 플레이어 입력, 동일한 액션 순서를 반복해도 `새 reset 이후에는` 자동 반응 결과가 달라질 수 있어야 한다.
- 기본 플레이에서는 결과가 항상 같아지는 고정 시드를 사용하면 안 된다.

### FR-5. 저장/복원 시 일관성 유지

- 한 에피소드 중간에 저장 상태를 다시 읽어도, 이미 생성된 Live Seed와 RNG 진행 상태는 유지되어야 한다.
- 즉, 한 에피소드 내부에서는 `현재까지의 상태를 기준으로` 일관되게 이어져야 한다.

## 비기능 요구사항

### NFR-1. 성능

- Autonomy Phase는 LLM 호출 없이 동작해야 한다.
- 한 턴 추가 비용은 체감상 거의 없어야 한다.
- 목표는 기존 턴 처리 시간에 비해 작은 증가만 허용하는 것이다.

### NFR-2. 가독성

- 한 파일에 planner, RNG, delta apply, event text를 모두 몰아넣지 않는다.
- 정책 데이터와 실행 로직을 분리한다.
- `Math.random()`을 직접 호출하지 않는다.

### NFR-3. 문서화 친화성

- public function 이름은 용어와 일치시킨다.
- 숫자 상수는 반드시 config/data 테이블로 분리한다.
- 디버그용 출력 구조는 추후 문서/운영 가이드에 옮길 수 있게 JSON-friendly shape로 만든다.

## 랜덤 정책

### 기본 원칙

- 기본 플레이에서는 `결과가 달라져야 한다`.
- 따라서 매 reset마다 새로운 `Live Seed`를 생성한다.
- 한 에피소드 내부에서는 그 seed를 고정하고 `drawCount`를 증가시키며 사용한다.

### 구현 원칙

- Live Seed 생성은 `crypto` 기반으로 한다.
- RNG는 wrapper를 통해서만 접근한다.
- business logic에서 `Math.random()` 직접 호출 금지

### 디버그 모드

- 테스트와 검증을 위해 `NPC_AUTONOMY_DEBUG_SEED` env를 지원할 수 있다.
- 단, 기본값은 `null` 이어야 한다.
- 기본 플레이 흐름에서는 이 env를 사용하지 않는다.

### 결과 다양성에 대한 명시 요구

- 동일한 입력 시퀀스를 새 에피소드에서 30회 반복했을 때 autonomy 결과 시퀀스가 모두 같아서는 안 된다.
- “무조건 다 다를 것”까지는 요구하지 않지만, `항상 똑같음`은 실패다.

## 턴 처리 흐름 변경

기존 흐름을 아래처럼 바꾼다.

### 기존

1. 플레이어 턴 반영
2. pressure update
3. round progress
4. resolve

### 변경 후

1. 플레이어 턴 반영
2. player-origin pressure update
3. round progress
4. round event append
5. `Autonomy Phase` 실행
6. autonomy-origin pressure update
7. resolve
8. inspector / log / export 정리

### 이 순서를 택하는 이유

- 새 라운드 이벤트가 공개된 직후 NPC들이 그 사건에 반응하는 구조가 더 자연스럽다.
- “구조 ETA 지연”, “문서 발견” 같은 라운드 이벤트가 autonomy planning에 바로 영향을 줄 수 있다.

## Autonomy planner 설계

### Step 수 결정

- 기본은 1 step
- 아래 조건에서 2 step 확률을 올린다.
  - round 4 이상
  - 현재 보드 1위와 2위 gap이 작음
  - 최근 round event tone이 `warning` 또는 `danger`

### Actor 선택

actor 후보는 기본적으로 `플레이어 제외`, `해당 턴에 이미 직접 반응한 request.npcId 제외`로 시작한다.

actor weight 계산에 반영할 요소

- `emotion.intensity`
- `decision.initialTargets`
- `relationship.npcOpinions`
- 현재 consensus leader와의 거리
- round event tag 일치도
- 현재 라운드 volatility

### Move 종류

V1은 아래 4개만 구현한다.

1. `pile_on`
   - 이미 위험한 대상을 조금 더 민다.
2. `shield`
   - 내가 덜 잃고 싶은 대상을 조금 감싼다.
3. `redirect`
   - 한쪽 책임선을 다른 NPC 쪽으로 조금 돌린다.
4. `freeze`
   - 큰 압력 이동은 만들지 않지만, opinion / emotion만 약간 조정한다.

### Target 선택

`pile_on` target 가중치

- actor의 `initialTargets`
- actor의 `npcOpinions[target]`가 낮을수록 가중치 증가
- 현재 보드 상위권일수록 가중치 증가
- 관련 round event tag와 연결되면 가중치 증가

`shield` target 가중치

- actor의 `npcOpinions[target]`가 높을수록 가중치 증가
- utility가 높은 캐릭터일수록 증가 가능
- actor와 role alignment가 있으면 증가

`redirect` target 가중치

- 현재 보드 1위가 actor가 선호하지 않는 대상이 아닐 경우, 2~3위 쪽으로 책임선을 살짝 옮기는 형태 허용

## Autonomy apply 설계

### 중요 원칙

- 기존 `applyInteractionPressure()`를 그대로 재사용하지 않는다.
- player-turn 규칙과 autonomy-turn 규칙은 별도 진입점으로 분리한다.
- 다만 아래 low-level helper는 재사용 또는 공통화할 수 있다.
  - `updateJudgementDimensions`
  - `buildConsensusBoard`
  - clamp helper

### delta 규모

autonomy는 플레이어 턴보다 약해야 한다.

권장 총량

- step 1회당 target 총 압력 변화: 대략 `4 ~ 12`
- 큰 경우에도 `20`을 넘지 않도록 제한

권장 dimension 예시

- `pile_on`
  - blame `+1 ~ +3`
  - distrust `+1 ~ +2`
  - hostility `0 ~ +2`
- `shield`
  - blame `-1 ~ -2`
  - distrust `-1 ~ -2`
  - sympathy `+1 ~ +2`
  - utility `+1 ~ +2`
- `redirect`
  - old target에 소폭 완화
  - new target에 blame/distrust 소폭 증가
- `freeze`
  - judgement는 거의 그대로
  - opinion / emotion만 소폭 변경

### npcOpinions 반영

V1에서는 `npcOpinions`를 아래 용도로 사용한다.

- planner의 target 선호도 계산
- apply 시 actor의 target opinion을 소폭 증감

예시

- `pile_on`: actor -> target opinion `-2 ~ -5`
- `shield`: actor -> target opinion `+1 ~ +3`
- `redirect`: actor -> oldTarget `+1`, newTarget `-2`

## 이벤트 로그 설계

### 표출 방식

- `EventLogEntry`만 추가한다.
- `InteractionLogEntry`는 추가하지 않는다.
- V1에서 autonomous text를 conversation UI에 섞지 않는다.

### 형식

- title: `방 안의 후속 반응`
- detail: 짧고 설명 가능한 문장
- tone:
  - pressure 증가 위주면 `warning`
  - 방어/완화면 `info`
  - 결말 직전 큰 흔들림이면 `danger`

### 예시 문구

- `박도현이 다시 서진호 쪽 책임선을 밀었다.`
- `한유리가 박도현을 바로 희생양으로 밀지는 않겠다는 태도를 보였다.`
- `마야 로웰이 책임선이 다시 흩어지지 않도록 감독관 논리로 정리하려 들었다.`

## Inspector / 디버그 데이터 설계

Autonomy Phase 결과는 다음 정보를 남긴다.

- step count
- 각 step의 actor
- move type
- target
- applied delta
- RNG draw summary
- phase 전후 consensus board top 3

이 데이터는 기본 UI에 전부 노출하지 않아도 된다.
하지만 payload에는 넣어서 향후 문서화와 밸런싱 분석에 활용한다.

## 테스트 계획

### 단위 테스트

반드시 분리 테스트한다.

1. `random.ts`
   - 같은 debug seed면 같은 draw sequence
   - live seed가 다르면 다른 draw sequence

2. `planner.ts`
   - player 제외 규칙
   - request.npcId 제외 규칙
   - target이 자기 자신이 되지 않음
   - disabled 시 빈 결과

3. `apply.ts`
   - 수치 clamp 보장
   - judgements 갱신 후 consensus board 계산 가능
   - step가 없어도 상태 보존

### 통합 테스트

1. 동일한 scripted turns를 30회 reset 후 반복 실행
2. 아래를 기록한다.
   - finalRound
   - resolutionType
   - sacrificedLabel
   - autonomy step sequence
3. 다음을 만족해야 한다.
   - autonomy step sequence가 1종류로 고정되지 않는다.
   - finalRound가 최소 2개 이상 나온다.
   - `minRoundsBeforeResolution` 이전 종료는 없어야 한다.

### 스모크 테스트

기존 `backend/scripts/smoke-episode.mjs`에 autonomy 검증을 추가한다.

추가 검증 항목

- 적어도 한 턴 이상 autonomy event가 발생했는지
- autonomy로 인해 event log가 비지 않는지
- resolved 여부와 export 흐름이 깨지지 않는지

## 코드 품질 제약

## 문서화 후속 작업 고려사항

이 기능은 구현 후 아래 문서로 파생될 수 있어야 한다.

- 사용자 설명용: “왜 매번 결과가 조금씩 다른가”
- 운영자 설명용: “autonomy phase는 어떤 규칙으로 움직이는가”
- 개발자 설명용: “player turn과 autonomy phase의 경계는 어디인가”
- 밸런싱 문서용: “어떤 수치를 어디서 조정하는가”

따라서 구현 시 다음을 지킨다.

- 용어를 코드와 문서에서 동일하게 쓸 것
- 이벤트 텍스트는 지나치게 서사적이지 말 것
- step 결과는 JSON으로 읽기 쉬운 구조를 유지할 것
- config와 결과를 diff하기 쉽게 만들 것

## 최종 승인 기준

아래를 만족하면 V1 완료로 본다.

1. 새 에피소드에서 같은 플레이를 반복해도 autonomy 결과가 고정되지 않는다.
2. 한 턴마다 방 안이 약간 더 살아 움직이는 느낌이 난다.
3. 플레이어 턴의 영향력이 autonomy보다 여전히 크다.
4. 종료 타이밍이 다시 과도하게 빨라지지 않는다.
5. 코드가 `planner / random / apply / event` 역할로 분리되어 읽힌다.
